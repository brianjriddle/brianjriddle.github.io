17 Mar 2010

today more physics, TDD concepts and tools, adding new functions to hunt the wumpus. The first part of the morning we went over coverage and metrics. Here we looked at emma, crap4j and discussed The Braithwaite Correlation.

The second part of the morning we looked at TDD vs BDD, GUI testing, legacy code, and convincing others to use TDD.

TDD vs BDD came down to semantics and readability. Sure there are lots of devil in the detail but it comes down to a matter of style.

GUI testing is not easy to do but by making direct calls to the APIs the GUI depends on without the GUI and testing the GUI without having the system behind it up (by stubbing it out). You can get the GUI tested pretty well. You will still need to test the entire system but by separating the testing of the business rules and the GUI you minimize the amount of manual and fragile testing. As the GUI can change much faster than the business rules this works well.

With legacy code a good start is reading working effectively with legacy code. Legacy code defined as code with out test is 1) notoriously hard to test 2) get under test 3) going to take a long time to get good code coverage. You really need to practice the boy scout rule. That means always leave the code better than when you found it.

Convincing others to use TDD/BDD/testing at all is really really hard. There is pretty much no point in trying to convince people that aren’t will to listen. What you can do is set an example as a professional by using TDD.

The afternoon was the practical where we got to start to add internalization to the hunt the wumpus game.

The physics covered to day was how white dwarves form, black holes and the big bang.

I would really recommend anybody that is practicing TDD or has been to a beginner TDD course. I have really enjoyed the course and am glad to have the chance. It’s going to take a couple of days for the full force of it to sink in but just the inspiration kick I have gotten is worth it.
