16 Mar 2010

Today more physics and theory. The physics covered the speed of light and Einstein’s special and general relativity. TDD theory was about what makes a well behaved unit test ie F.I.R.S.T, test conventions (where to place them, how to name them). Where to place them unclebob likes to place them right next to the files being tested but most conventions (ie maven) place them in a separate directory. This is kinda a moot point as modern java editors can present a combined view of tests and packages. How to name them an interesting way that unclebob is using now is METHOD_WHEN_THEN.

We spent some time discussing testing private and protected methods. Unclebob uses the following rules(guides) concerning testing private/protected methods

  • don’t test private or protected
  • testing trumps privacy
  • testing privates implies a design error

As a rule he does not generally test private methods as the new method is public to begin with and after he adds tests/functionality to the public method he refactors out most of the public method so all the private methods are covered by test as the tests against the public methods exercise the refactored private methods.

I had a little problem with starting with adding new functionality in a public method but I believe it is mainly because I have dealt with legacy code and have had as a rule to not change the interface of the class being changed. When dealing with legacy code I think that starting with package protected is fine as the code I have been working in has had 0 to few tests so that changing the interface and adding a method that can be called outside of a specific scope is tough. By doing this I am able to keep the interface stable while adding the ability to get code under test. When and if the method becomes needs to be called in other packages or members of the class hierarchy it is then that I open up the methods.

The last part of the day we started to cover fitness by correcting and implementing acceptance tests.
